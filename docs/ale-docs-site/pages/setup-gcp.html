Set up Google Cloud

One-time provisioning to run ALE on Google Cloud: a project, a service account, the two sandbox images, networking, and a results bucket. About 10 minutes, mostly waiting on the image copies. Each step below explains what it sets up and why. If you have done this before and just want the commands, docs/quickstart.md in the repo is the condensed version of this page.

Create a Google Cloud account and install the gcloud CLI

Skip the account step if you already have one. Open cloud.google.com/free, click Get started for free, sign in with a Google account, pick your country, accept the terms, and add a payment method.

A credit card is required, but only to confirm you are a real person. On the free trial Google never charges it. You get $300 of credit for 90 days; if you use it up (or the 90 days pass), your resources simply stop. The card is billed only after you deliberately activate a full, paid billing account and then spend beyond whatever credit remains.

Free trial runs Linux only
The trial can create Linux VMs but not Windows ones. The demo (demo/hello) is Linux, so it runs on the trial without ever touching your card. The full benchmark is mostly Windows, so to run it you must activate a full billing account (Console → Billing → Activate full account; your $300 credit still applies, and only spend beyond it bills the card).

Then install the gcloud CLI and sign in: 

gcloud auth login

Provision the project

① Create a project and turn on billing

Everything in Google Cloud lives inside a project: one billing and permissions boundary that holds your VMs, key, and storage. ALE works entirely inside a single project you own. Billing has to be linked even when you are spending free credit, otherwise the Compute and Storage APIs stay switched off.

First set a few shell variables that the rest of the page reuses. Only GCP_PROJECT is yours to choose — it must be unique across all of Google Cloud, so change it if creation says the ID is taken. Leave the rest as they are. 

export GCP_PROJECT="ale-$(whoami)"          # must be globally unique
export GCP_REGION="us-central1"             # default; fine to leave as is
export GCP_SA_NAME="ale-runner"
export GCP_SA_EMAIL="${GCP_SA_NAME}@${GCP_PROJECT}.iam.gserviceaccount.com"
export GCP_BUCKET="${GCP_PROJECT}-ale-results"

Then create the project, link billing, and turn on the APIs:

gcloud projects create "$GCP_PROJECT" --name="ALE"
gcloud config set project "$GCP_PROJECT"

gcloud billing accounts list                       # shows your billing-account IDs
read -p "billing-account ID from the list above: " BILLING_ID
gcloud billing projects link "$GCP_PROJECT" --billing-account="$BILLING_ID"

gcloud services enable compute.googleapis.com storage.googleapis.com

② Create a service account for Cloud Storage

The service account's JSON key is copied into every gcloud VM sandbox a run creates, where the VM's gsutil uses it to pull the current task's data from a GCS bucket and to sync the agent's output artifacts back to your results bucket (the recommended path, see output_path). The two roles cover exactly that: read objects, and bill requester-pays reads to your project. The key is written to secret/gcp_key.json, which the environment config points at on the next page. 

gcloud iam service-accounts create "$GCP_SA_NAME" --display-name="ALE storage access"

for role in storage.objectViewer \
            serviceusage.serviceUsageConsumer ; do
  gcloud projects add-iam-policy-binding "$GCP_PROJECT" \
    --member="serviceAccount:$GCP_SA_EMAIL" --role="roles/$role" --condition=None
done

mkdir -p secret
gcloud iam service-accounts keys create secret/gcp_key.json --iam-account="$GCP_SA_EMAIL"

③ Copy the sandbox images into your project

A task boots from a prebaked disk image: an OS with the professional software and ALE's in-VM tooling already installed. ALE publishes two, ale-ubuntu22 (Linux) and ale-win10 (Windows). Copy them once into your own project so VMs boot from your copy and do not depend on ours at run time. Each copy takes a few minutes. 

for img in ale-ubuntu22 ale-win10 ; do
  gcloud compute images create "$img" \
    --source-image="$img" --source-image-project=agenthle-488519
done

④ Open the network path to the sandbox

Each VM comes up on a VPC network with its own IP. ALE reaches the in-VM control server over tcp:5000 to launch the agent and pull artifacts, so the firewall has to allow that port. RDP (tcp:3389) is optional, open it only if you want to watch a Windows desktop live. 

gcloud compute networks create ale-vpc --subnet-mode=auto

gcloud compute firewall-rules create ale-allow-cua --network=ale-vpc \
  --direction=INGRESS --allow=tcp:5000 --source-ranges=0.0.0.0/0
gcloud compute firewall-rules create ale-allow-rdp --network=ale-vpc \
  --direction=INGRESS --allow=tcp:3389 --source-ranges=0.0.0.0/0

⑤ Create a results bucket (recommended)

The Cloud Storage bucket each VM syncs its output to. Create it and grant the service account write access; you point output_path at it on the next page. 

gcloud storage buckets create "gs://$GCP_BUCKET" \
  --project="$GCP_PROJECT" --location="$GCP_REGION" --uniform-bucket-level-access
gcloud storage buckets add-iam-policy-binding "gs://$GCP_BUCKET" \
  --member="serviceAccount:$GCP_SA_EMAIL" --role="roles/storage.objectAdmin"

Every command is idempotent except gcloud projects create, which errors if the project already exists. If a step fails, fix it and re-run just that step.
