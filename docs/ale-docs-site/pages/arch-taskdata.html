Task spec & data staging

Curated real-world inputs, a hidden reference answer, and a grader that can't be gamed. The data is what makes an ALE task more than a prompt — and keeping the answer out of the agent's reach is a first-class design concern.

A task is a two-file package: a card of metadata plus load / start / evaluate hooks. The full schema is for authors (see Add a task); what matters architecturally is the data a task moves onto the sandbox and how that data is graded — the subject of this page.

What a task carries

Beyond its instructions, a task ships two kinds of data, collected and validated by domain experts so that the inputs plus the prompt determine a gradeable result:

KindVisible to the agent?Purpose
inputYes — staged before the runThe materials the task starts from.
referenceNo — the answer keyThe ground truth the grader scores against.

(A few tasks also drop in a small amount of task-specific software, but most rely on what the image already provides.)

Where the data comes from

The data can reach the VM two ways, chosen by the environment's task_data_source:

  • Baked into the image — the default. Inputs and a reference archive already sit on disk; nothing is fetched at run time.
  • Pulled per task from a public bucket — point task_data_source at a gs:// prefix and each task syncs just its own data when it runs. This is how someone building their own sandbox gets the data without rebaking an image. (A Hugging Face source is stubbed for the future.)

Keeping the answer hidden

A real risk with output-graded tasks: an agent that goes looking could simply read the reference off the disk. ALE blocks that two ways at once.

Encryption at rest. A baked-in reference ships as a password-protected archive (reference.7z), opaque while the agent is working. It is decrypted only at grading time. The password is a shared barrier — enough to stop the agent reading the key in passing, not a cryptographic secret.

Timing. The reference isn't even present during the run. Inputs are staged before the agent starts; the reference is staged only after it finishes, in a dedicated phase, and the bucket path wipes and re-syncs it fresh so no partial state lingers. Tasks assert the reference is unreadable during setup and fail loudly if it leaks.

Grading

Once the reference is available, the task's evaluate hook reads the agent's output alongside it and returns a list of scores. Two styles of grader are used:

  • Deterministic — exact or structural comparison (files, values, parsed content). Most tasks.
  • Judge-based — an LLM or VLM scores outputs that need semantic or visual judgment (e.g. an OpenAI vision judge, a Gemini video judge). A task declares which judge credentials it needs in its evaluatorCredentials; the keys live in secret/eval_time/ and load automatically at grading time.

The grader runs as part of the task slot, host-side, after the agent is gone — so even judge-based scoring never touches the agent's run.
