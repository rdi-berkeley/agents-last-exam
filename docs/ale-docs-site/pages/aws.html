Set up AWS

One-time provisioning to run ALE on AWS: credentials, networking, the two sandbox images as AMIs, and S3 buckets for task data and results. Linux is a few minutes plus the image step; Windows needs more (see the Windows note below). If you just want the commands, docs/aws-setup.md in the repo is the condensed version, and tools/aws/ holds the scripts this page runs.

Linux is easy; Windows is not
Linux (the demo/hello task and all cpu-free-ubuntu tasks) runs on ordinary shared EC2 with no special licensing. Windows 10 on EC2 is a desktop/client OS: AWS sells no license for it, so it is BYOL (you supply a Microsoft VDA E3/E5 subscription) and must run on dedicated tenancy (a Dedicated Instance, raised via an AWS Support case). Start with Linux; tackle Windows once you have those.

Create an AWS account and install the AWS CLI

Skip the account step if you already have one. Open aws.amazon.com, choose Create an AWS Account, and complete sign-up (email, a payment card for identity verification, phone verification, the Basic/Free support plan). Unlike Google Cloud there is no blanket free trial credit; you pay for what you use. Getting Linux working end to end (importing one image, booting a test instance for a few hours) costs roughly a few US dollars, mostly EBS snapshot storage for the ~600 GiB disk.

Then install the AWS CLI v2 and give it credentials. Create an IAM user with programmatic access (Console → IAM → Users → Create user; attach AdministratorAccess for setup, scope down later), make an access key for it, then: 

aws configure           # paste the Access Key ID + Secret, region us-east-1, output json
aws sts get-caller-identity   # confirms the CLI is authenticated

Provision the account

① Bootstrap networking, the import role, and the buckets

One script creates everything a run needs: the vmimport service role (lets AWS convert disks into AMIs), a VPC with one public subnet per Availability Zone (the provider tries them in order on capacity errors, like gcloud zones), a security group that opens the in-VM control port (tcp:5000 for cua-server, tcp:3389 for Windows RDP), and an S3 import bucket. It writes the resolved ids to tools/aws/.aws-env, which later steps and the environment config read. 

export AWS_REGION=us-east-1
AWS_REGION=$AWS_REGION ./tools/aws/bootstrap_account.sh
source tools/aws/.aws-env          # exports ALE_AWS_VPC, ALE_AWS_SG, ALE_BUCKET, ...

② Get the sandbox images (AMIs)

A task boots from a prebaked disk image: an OS with the professional software, ALE's in-VM tooling, and the task data already baked in. ALE has two families, ale-ubuntu22 (Linux) and ale-win10 (Windows) — the same family names the Google Cloud setup uses. On AWS each family resolves to an AMI (an Amazon Machine Image, region-scoped). You have two ways to get one:

(a) Use a published public AMI (Linux, if available in your region). AWS AMIs can be made public, so others launch them directly with no copy — as long as they are in the same region. If ALE publishes ale-ubuntu22 in your region you skip importing entirely; for a different region you copy it once (aws ec2 copy-image). This is lighter than gcloud, where every image is always copied into your own project.

(b) Import it yourself (works in any region; required for Windows). ALE exports the disks to gs://ale-data-public/images/. The scripts stream a disk into your S3 import bucket and register an AMI from it: 

# Linux — import-snapshot + register-image (boots on Nitro; ~1.5-4 h to convert)
./tools/aws/import_ubuntu_ami.sh

# Windows — BYOL import (you must hold a Microsoft VDA license; ~2-5 h)
./tools/aws/import_win10_ami.sh

Each script tags the resulting AMI with ale:image-family=<name>. That tag is the link the provider uses: at run time it looks up the newest AMI carrying the family tag, so the environment config just names the family (image: ale-win10) and never a raw ami-… id. To swap in a rebuilt image later, tag the new AMI the same way (newest wins) or pin one explicitly with ami: in the config.

Why Windows must be imported, not shared
A public Linux AMI is fine. A Windows 10 AMI cannot be shared for others to use: Windows client licensing is per-account BYOL, so every user imports their own from the disk under their own Microsoft license. The import is also why Windows needs the vmimport role from step ①.

③ Task data and results buckets (S3)

Task data reaches the sandbox one of two ways. The simplest, and the default for the images above, is baked into the AMI — the disk already carries input/ and the encrypted reference for every task, so nothing is fetched at run time and you can skip this step (task_data_source: baked_in_sandbox).

The alternative mirrors gcloud's gs://ale-data-public bucket: stage data from S3 at run time (task_data_source: s3://<bucket>). For that you copy the task-data bucket into S3 once — the on-disk layout and the encrypted 7z reference are unchanged, only the scheme differs: 

# one-time, host-side (rclone streams gs:// -> s3://; ~hundreds of GB)
rclone copy gcs:ale-data-public s3:<your-data-bucket> --transfers 16

Results work the same as gcloud's results bucket. Point output_path at s3://<your-results-bucket> and each sandbox pushes its output there with the in-box aws CLI (the AWS analogue of gcloud's gsutil sync to ale-run-results).

Both the s3:// data pull and the s3:// output push run inside the sandbox, so the instance needs S3 permissions. On AWS that is an IAM instance profile attached at launch — no key is copied into the box (unlike gcloud, which injects a service-account key). Create a role with read on the data bucket and write on the results bucket, then set iam_instance_profile: on the snapshot. If you only use baked data and don't push output, you can skip the profile.

④ Windows only: dedicated tenancy and GPU quota

Two account limits gate Windows and GPU, both raised through AWS Support / Service Quotas (a new account starts at or near zero on each, and approval can take hours to days):

  • Dedicated tenancy — required for Windows 10 BYOL. Open a Support case requesting Dedicated Instances for your region. The environment config already launches Windows snapshots with tenancy: dedicated; nothing else changes once it's granted.
  • GPU instances — the gpu-* snapshots use a g6 instance (NVIDIA L4, matching the image's driver). The "Running On-Demand G and VT instances" quota defaults to 0; request an increase (it's self-service in Service Quotas). Whether the baked NVIDIA driver binds on AWS GPUs is verified separately.

⑤ Point the environment config at it

configs/environments/environment_aws.yaml is the AWS counterpart of environment.yaml. It reads AWS_REGION, ALE_AWS_VPC, and ALE_AWS_SG from your shell (set by step ①), names each image by family, lists Availability Zones in zones, and marks the Windows snapshots dedicated. Start with one Linux task to confirm the path end to end: 

source tools/aws/.aws-env
python -m ale_run ... --environment configs/environments/environment_aws.yaml   # task_data_source: baked_in_sandbox
