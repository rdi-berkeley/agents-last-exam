Sandbox & provider

The environment slot is where a task runs: a sandbox, provisioned on demand. This page covers what a sandbox is, how providers bring one up, and the single surface every other layer uses to reach into it.

A sandbox is a whole computer

ALE's unit of environment is a sandbox: a complete Windows or Linux operating system, with the real applications installed and the task's files on disk — a desktop the agent actually operates. 感觉这里原来那段没啥意义，

The SandboxHandle: one way in

Everything that touches the sandbox — staging task data, installing an agent, a task's grader — goes through the SandboxHandle the provider returns. It is a thin async surface (run_command, read_file, write_file, uploads, ranged downloads) plus the sandbox's coordinates: its endpoint, its OS, and the image's path conventions. No layer reaches the sandbox any other way. 这里应该说一下我们使用trycua 【https://github.com/trycua/cua】提供的 cua-server 来给不同系统提供一个统一的远程控制接口，并基于此套接口去manipulate computers。give them credits for providing such a powerful and easy to use computer control package.

cua: the control server inside the sandbox

The handle isn't magic — it's an HTTP client. Each image runs a cua computer-server (from the open-source trycua/cua project): a small server that exposes the sandbox's shell, filesystem, and GUI over HTTP. The SandboxHandle and the task-side session are both clients of it.

framework host sandbox ┌───────────────┐ ┌─────────────────────────┐ │ SandboxHandle │ ───── HTTP /cmd ─────▶ │ cua computer-server │ │ (+ session) │ ◀──── results ──────── │ shell · files · GUI │ └───────────────┘ └─────────────────────────┘

open_session() returns a RemoteDesktopSession — the richer client task authors use during setup and grading (screenshots, clicks, window control, in-sandbox Python). Its full surface is the docs/SESSION_API.md. The server's port is declared per image (5000 on the GCE images, 8000 on the kasm container), so nothing in the framework hard-codes it.

The image: what's already on the sandbox

A provider boots from an image, and the image is where realism is pre-paid. Published images carry the OS, the cua server, the agent CLIs that run in-sandbox, the Node and Python runtimes, the GUI MCP bridge, and — for many tasks — the task data itself. An image registry (environments/images/) records each image's paths and port so the rest of the framework stays image-agnostic.
