Configure & run a benchmark

With the cloud provisioned, a run is just three config files wired together: an experiment picks which agents, which environment, and which tasks. This page walks each one, the two kinds of API key, and the launch command.

The experiment file

An experiment only wires things together — it references the agent and environment configs by path and carries run-level knobs. Copy example_exp.yaml and edit: 

name: ale_run                                   # names the output folder

secret_file: secret/.env                        # API keys (loaded before parsing)

agents:                                         # one path per agent — list several to run a matrix
  - configs/agents/claude_code.yaml
environment: configs/environments/environment.yaml
tasks: selected_tasks/unlicensed/near-term.txt  # a task list (a tier, here)

output:
  root: .logs                                   # host-side run logs
concurrency: 8                                  # units (agent × task) in flight at once
cleanup_mode: delete                            # delete | stop | keep the VMs after each run
FieldWhat it does
agentsPaths under configs/agents/. Several entries → every agent runs over every task.
environmentExactly one path under configs/environments/.
tasksA task-list file under selected_tasks/ — see Configure the tasks below.
concurrencyHow many agent×task units run at once — caps live VMs and LLM rate.
cleanup_modedelete for batch runs; keep to leave a VM up for debugging.

Configure the environment

A task declares which snapshot it needs in its task card, and the environment config (configs/environments/environment.yaml) maps each snapshot to a backend — provider, image, and connection. ALE's tasks span five snapshots:

SnapshotOS · GPU · licenseImage
cpu-free-ubuntuLinux — the bulk of tasksale-ubuntu22
cpu-freeWindowsale-win10
cpu-licenseWindows · licensedale-win10
gpu-freeWindows · GPUale-win10
gpu-licenseWindows · GPU · licensedale-win10

Every Windows variant folds onto a single image (ale-win10), so two images cover all five. Leave the snapshots: block exactly as shipped — it already maps each snapshot to the right image and pulls your project and key in through ${env:…}. (As you add your own providers or images, configure these however you like.)

The fields below the snapshots are the ones worth knowing:

task_data_source: baked_in_sandbox    # task data is baked into the images — nothing to fetch
gcs_sa_key: secret/gcp_key.json       # the SA key from GCP setup — already in place
output_path: local                    # ← the one field to change for batch runs (below)
output_path — strongly prefer the bucket

local pulls each task's output off the sandbox over its cua computer-server (port 5000) — slow and, with large artifacts, fragile on a remote VM, so not recommended there. (Natural for a local sandbox, handy once the VMware / QEMU providers land.)

The GCS bucket uploads VM-side, straight to storage — faster and robust at scale, and already created during GCP setup: 

output_path: gs://<GCP_PROJECT>-ale-results

Each run lands at gs://<GCP_PROJECT>-ale-results/<run_id>/output/, keyed by run id — pull any run back with gsutil -m cp -r on its prefix, or script a whole sweep.

Configure the agents

Every agent ships a template under configs/agents/ with each knob set to its default, so the tunable surface is visible at a glance. A preset is three things — the harness, the model, and an agent-specific config block: 

# configs/agents/claude_code.yaml
harness: claude_code
model: anthropic/claude-sonnet-4.6
config:
  provider: openrouter        # openrouter → OPENROUTER_API_KEY ; direct → ANTHROPIC_API_KEY
  max_turns: -1               # -1 = unlimited
  # … agent-specific knobs, all shown at their defaults

The config fields differ per agent — that's why each has its own template. List several agent files under the experiment's agents: to sweep a matrix (every agent over every task).

Configure the tasks

tasks: points at a list under selected_tasks/:

ListWhat it is
full/{near-term,full-spectrum,last-exam}.txtThe full benchmark, split by difficulty tier.
unlicensed/…The same tiers minus tasks needing licensed software — runs on the published images as-is.
linux_only.txtThe Linux-only subset (no GUI desktop).
helloworld.txtA single demo task.

Tiers: near-term (cheapest, partly solvable today), full-spectrum (one task per subdomain), last-exam (the hardest). A .txt list is one task path per line; a .yaml list adds per-task variant control.

Keys: agents vs judges

Two separate sets of keys, for two separate jobs.

  • Agent keys — secret/.env. Used to run the agents. The reference presets route through OpenRouter, so set OPENROUTER_API_KEY. A provider's official key (ANTHROPIC_API_KEY, …) is only used when an agent's config sets provider: direct. 
    OPENROUTER_API_KEY=sk-or-...
GCP_PROJECT=<your project>             # read by the env config (${env:GCP_PROJECT})
GCP_SA_KEY=secret/gcp_key.json         # read by the env config (${env:GCP_SA_KEY})
  • Judge keys — secret/eval_time/. Used to grade. Some tasks score their output with an LLM/VLM judge; those keys live per-service here and load automatically at scoring time. Copy the template and fill it: 
    cp secret/eval_time/openai.env.example secret/eval_time/openai.env   # then add the key
    Required for judge-scored tasks — without it those tasks can't be evaluated. A task's task_card.json evaluatorCredentials names which file it needs.

Run it

uv run python -m ale_run run my_exp.yaml --dry-run   # validate the agent × task matrix
uv run python -m ale_run run my_exp.yaml             # run it

--dry-run prints the matrix without booting anything. The real run boots sandboxes (bounded by concurrency), runs each agent, grades it, and tears the VMs down. Each unit writes a scored, replayable record — its layout is covered in Trajectories & artifacts.
