Add an environment

A new backend — another cloud, a hypervisor, your own hardware — is a Provider that can produce a live sandbox the framework can drive. See Sandbox & provider for the concepts.

Implement the Provider contract

class MyProvider(Provider):
    async def acquire(self, spec: SandboxSpec) -> SandboxHandle: ...   # bring one up
    async def release(self, sandbox, *, mode="delete"): ...           # delete | stop | keep
    def open_session(self, sandbox) -> RemoteDesktopSession: ...       # a live cua session

    async def heartbeat(self, sandbox): ...          # optional: refresh leases mid-run
    async def cancel_external(self, sandbox): ...    # optional: kill in-flight ops on SIGINT
MethodDoes
acquire(spec)Boot a sandbox for spec (its snapshot, OS, sizing, GPU) and return a populated SandboxHandle.
release(…, mode)Tear down per mode. Honor keep — debugging relies on it.
open_session()Return a RemoteDesktopSession wired to sandbox.endpoint; the framework threads it into the task hooks.

The SandboxHandle must carry id, endpoint, os, and the VM-side paths (work_dir_base, task_data_root, node, python, mcp_server_dir) — inherit them from the image registry via image.sandbox_paths().

Register it

Add the provider to the dispatch in orchestration/factory.py, then ship an env config that names it plus its knobs; an experiment wires it in by path: 

# configs/environments/my_provider.yaml
provider: my_provider
region: us-west2
# … snapshots: { ... }  (snapshot → image map, as in environment.yaml)

Images

If your sandbox bakes things at the standard paths, reuse an existing image family from environments/images/; otherwise add one (declaring os, the work/data roots, and the node / python / mcp_server_dir paths).

Staying on GCP? Just swap the image
To run a different OS or toolset on the existing provider, you don't need new code — point a snapshot's image: at your custom image in the env config. Tasks asking for that snapshot boot it instead.
