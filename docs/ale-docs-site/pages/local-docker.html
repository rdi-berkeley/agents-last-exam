Local Docker

Run the Linux subset in a container on your own machine — no GCP, no VM. Experimental and under active validation.

Experimental
The image and provider wiring work, but task coverage is not yet validated. The sandbox images were authored for a full VM; a container shares the host kernel and runs no init system, so some tasks may not be completable here. See Open risks before relying on results.

How it works

The docker provider boots ale-ubuntu22-docker, a container image whose userspace is a byte-for-byte export of the ale-ubuntu22 GCE VM's root filesystem — the same node, Python venv, cua-server and pre-installed task software, at the same absolute paths. On acquire it starts one container per run unit, brings up cua-server on port 5000 behind a virtual X display (Xvfb :0), and drives it over the same cua HTTP API as a cloud VM. Task data is staged from gs://ale-data-public with the service-account key the provider injects per container.

Build the image

The 275 GB image is built locally by exporting the VM rootfs (no package rebuild) and importing it as a single layer; a startup entrypoint and a few container-only fixups are baked on top. The build recipe is tracked in the umbrella PR — it is not yet a one-command script.

Configure a run

Point the Linux snapshot at the docker provider in your environment config, then configure a run as usual: 

snapshots:
  cpu-free-ubuntu:
    provider: docker
    image: ale-ubuntu22-docker   # container ref + cua port come from the Image entry
    docker:
      cpus: 4
      memory: 16g
      shm_size: 1g

Open risks

A container is not a VM. Tasks that assume a real kernel, an init system, privileged operations, kernel modules, a full desktop session, or hardware devices may behave differently or fail. Which tasks in the Linux subset survive the move is the open question this path is being validated against; expect gaps until that work lands.
