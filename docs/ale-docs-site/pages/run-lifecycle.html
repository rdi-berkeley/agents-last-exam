The run lifecycle

The clearest way into the architecture is to watch one run from start to finish. This page is that walkthrough — the whole system in motion. The rest of the section then takes each piece apart, and every phase below links to where it's decomposed.

From experiment to runs

python -m ale_run run <exp.yaml> loads the config — resolving the secret file and ${env:VAR} references — then pairs every agent with every task. Each agent–task pair is one run, and runs proceed with bounded concurrency. --dry-run lists them without booting anything; --agent and --task narrow the set.

The phases of a run

Every run goes through the same sequence. The environment tracks which phase it's in, and each transition is written to the event log — so a failure is always pinned to a phase.

① provision boot the sandbox, open a session ② stage inputs copy the task's input onto the sandbox; run task setup ③ agent run install + launch the agent; tail its logs as it works ④ evaluate stage the hidden reference, then grade the output → score ⑤ collect & record pull the agent's output and logs; build the trajectory ⑥ cleanup the provider releases the sandbox (delete / stop / keep)

① Provision

The provider boots a sandbox for the task's snapshot and opens a session. The resulting SandboxHandle is the I/O surface every later phase uses.

② Stage inputs & set up

The task's input data is staged onto the sandbox — from the image or a bucket, per the environment's data source — and the task's start hook builds the workspace. The hidden reference is deliberately not staged here.

③ Run the agent

The agent now runs to completion. An agent is packaged as a deployer — a small piece of code that knows how to install it, launch it on a prompt, and read its logs back — and an executor decides where that deployer runs: inside the sandbox, or beside it. The executor calls install() then launch(prompt) and waits, bounded by the task's time limit. For in-sandbox runs, the agent's logs are tailed off as it works, so an interrupted run still leaves a partial trace.

④ Evaluate

Now that the agent is done, the hidden reference is staged and decrypted — kept out of the agent's reach until exactly this point — and the task's evaluate hook scores the output against it, producing the run's score.

⑤ Collect & record

The agent's output files and its raw work directory are pulled off the sandbox and kept verbatim, and the deployer's parse_artifacts() reads the agent's native logs into a uniform trajectory. These steps and the grading above overlap in practice; their exact order isn't significant.

⑥ Cleanup

The provider releases the sandbox according to cleanup_mode: delete (default), stop, or keep for debugging.

What a run leaves behind

Each run writes a timestamped directory — its score, trajectory, event log, and the raw artifacts — detailed in Trajectories & artifacts. One property matters at the lifecycle level: the event log is appended and flushed line by line, so the trace survives even if a run is killed mid-phase.
