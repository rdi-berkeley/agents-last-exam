ALE · Open Evaluation Framework

Measure agents on real work, in real sandboxes.

Agents' Last Exam runs AI agents on long-horizon, economically valuable tasks inside full operating-system sandboxes, then grades what they produce against hidden references. This site explains how the ale_run framework works and how to run, configure, and extend it.

Why ALE is different

Most benchmarks simplify the world into a tidy API. ALE refuses to. Its tasks are real production workflows, so the evaluation environment has to be real too.

Real environments
Full Windows / Linux VMs with actual production software — not simulations.
Curated real data
Each task ships real input data and a hidden reference, collected and validated by domain experts.
Verifiable outcomes
Deterministic and judge-based graders score the agent's output against that reference.
Broad coverage
150 public tasks spanning 55 industries, from a 1,500+ task corpus.

How it works, in one minute

Frontier agents are whole systems — their own runtime, tools, memory, and sub-agents. ALE doesn't try to drive them step by step. It does the opposite: drop the agent into a real sandbox with the task, let it work to completion, then grade what it leaves behind. Afterwards it reconstructs a uniform trajectory from the agent's own logs, so every agent — however different inside — produces comparable records.

To make that swappable, ALE keeps three pieces independent:

┌────────────────────────────────┐ │ experiment.yaml │ │ agent + environment + tasks │ └────────────────┬───────────────┘ │ the orchestrator wires them together, │ runs the agent, then grades the result ┌────────────────────┼────────────────────┐ ▼ ▼ ▼ ┌────────────────┐ ┌────────────────┐ ┌────────────────┐ │ AGENT │ │ ENVIRONMENT │ │ TASK │ │ what to test │ │ where to run │ │ what to do + │ │ │ │ (the sandbox) │ │ how to grade │ └────────────────┘ └────────────────┘ └────────────────┘

Pick one of each in a YAML file, run one command. Swap any single slot and the other two are untouched — a new agent runs against every existing task, a new task runs on every existing environment.

How an agent plugs in

This is the idea worth internalizing early. An agent is added as a deployer: a small piece of Python that knows how to install the agent, launch it on a prompt, and read its logs back into a trajectory. The deployer never knows where it runs. A separate executor decides that — inside the sandbox, in a local container, or on your own machine.

The payoff
Write a deployer once; the same code runs in all three places. The agent's logic and the question of where it executes are kept completely apart.

What a run gives you

Each run writes a timestamped folder under .logs/: a final score, a structured trajectory (every step, tool call, and observation), an append-only event log, and the raw artifacts pulled off the VM.
